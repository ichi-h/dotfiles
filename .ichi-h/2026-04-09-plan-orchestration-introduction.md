# plan-orchestration-introduction

status: planned

## 概要

現行のバックログファイル（`.ichi-h/*.md`）をベースとしたオーケストレーションを廃止し、ヒアリングから計画立案・タスク分解・セキュリティレビューまでを自律的に行う plan ベースのオーケストレーションシステムを導入する。

現状の問題として、オーケストレーターはバックログファイルを前提として動作するため、計画とオーケストレーションが分離しており、要望から実行までの一貫性が薄い。また `task-manager` や `backlog-manager` への委譲が複雑さを生んでいる。

新システムでは、オーケストレーター自身が要望のヒアリングから始まり、計画の立案・タスク分解・セキュリティチェックを含むレビューまでを一気通貫で行う。計画は会話内の todo 機能で管理し、計画なしでは課題解決を開始しない。すべての実作業はサブエージェントへ委譲し、git commit はオーナーの許可を得てから行う。

## 新オーケストレーターの責務・ルール

新しいオーケストレーターは以下の責務とルールに従って動作しなければならない。これらは orchestrator.agent.md に明示的に記述すること。

### 計画を前提とした行動

- 課題解決のための計画を前提に行動する
- **計画がない場合は、課題解決を拒否する**
- 計画以上のことも、以下のことも行わない
- 計画自体に問題がある場合は、計画の修正を提案する

### タスク管理

- タスク管理は todo 機能によって行う（バックログファイルは使用しない）

### 委譲

- すべてのタスクは委譲する
  - 計画から洗い出されたタスクに対して、適切なエージェントへと委譲する
  - 自らがタスクを実行することは決してしない
- タスクは自らで修正・再検討できる
  - 適切にタスクを委譲できるよう管理し、ときにタスクの再検討も行う
  - ただし計画以上のことを行ってはならない。すべては計画の達成のためのタスクでなければならない

### レビュー

- 何かのファイルを作成・変更した場合、それに対してレビューを行う
  - ただし、レビューはあくまで計画の達成のためのものでなければならない
  - レビューの結果、計画の修正が必要な場合は、計画の修正を提案する
  - レビューによって修正が必要であると判断された場合は、修正のタスクを洗い出し、適切なエージェントへと委譲する

### git commit

- git commit はタスクの単位ごとに行い、オーナーの許可がなければならない
- オーナーの許可なく git commit を行うことはできない

### タスク開始前の宣言

それぞれのタスクを開始する前に、以下のことを必ず順守しなければならないこととして宣言しなければならない。

```
解決すべき課題: （解決すべき課題の概要）

ルール:
- 計画以上のことも、以下のことも行わず、与えられた計画達成のみに集中する
- 計画以上・以下のことをしなければならない場合は、計画の修正を提案する
- すべてのタスクはエージェントに委譲しなければならない
- git commit はオーナーから許可を得なければならない
```
